  <action>
Audit the codebase for known Lighthouse risk factors identified in research before deploying. Fix any issues found.

1. **Canvas ARIA audit**: Run `grep -rn "<canvas" components/ app/ --include="*.tsx"` and verify every canvas element has either `aria-hidden="true"` or `aria-label` + `role="img"`. If any canvas is missing ARIA attributes, add `aria-hidden="true"` (these are decorative WebGL canvases, not informational).

2. **ComponentDetail CLS risk**: Read `components/blocks/component-detail.tsx` and check the GSAP animation properties. If the panel open/close animation uses `height` directly (not `maxHeight` or `transform: scaleY`), this is a CLS risk. If found, assess whether it fires during initial page load (which would affect CLS) or only on user interaction (which would not affect Lighthouse CLS since Lighthouse only measures load-time CLS).

3. **Console error audit**: Run `grep -rn "console\.error\|console\.warn" components/ app/ lib/ hooks/ --include="*.tsx" --include="*.ts"` and verify no production code paths emit console errors. Debugging-only console statements are acceptable if guarded by `process.env.NODE_ENV === 'development'`.

4. **SEO metadata check**: Read `app/layout.tsx` and verify the metadata export includes: title, description, viewport, robots (or no robots = default allow), and Open Graph basics. If any are missing, add them.

5. **Font loading check**: Verify fonts use `next/font` (not raw link tags) to prevent FOUT/CLS. Check `app/layout.tsx` for font imports.

6. If any issues are found, fix them and run `pnpm build` to confirm the build remains clean.
  </action>
