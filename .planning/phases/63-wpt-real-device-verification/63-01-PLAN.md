<threat_model>
**Severity:** HIGH (block-on threshold per workflow.security_block_on=high)
**Surface:** WPT API key (`~/.wpt-api-key`, chmod 600) — sensitive credential.

### T-63-01: API key leak via stdout / log file / git commit
- **Vector 1 — stdout/stderr leak:** Inline shell substitution like `curl -d "key=$(cat ~/.wpt-api-key)" -v` echoes the key in verbose curl output and shell traces. Mitigation: NEVER use `echo`, `printf`, or `set -x` on any line that touches `$WPT_API_KEY`. NEVER pass the key as a request body field; pass via `-H "X-WPT-API-KEY: ${WPT_API_KEY}"` only. Disable curl `-v` flags in all production commands.
- **Vector 2 — log file leak:** `script`, `tee`, or shell history may persist key. Mitigation: do not pipe submission commands through `tee`; do not run with `script`; explicitly avoid writing the key to `/tmp/wpt-test-ids.txt` or any logged file. Test IDs (returned by submission) are safe to log; the key is not.
- **Vector 3 — git commit leak:** `~/.wpt-api-key` lives OUTSIDE the repo by design. Any temporary file created during processing must NEVER be staged. Mitigation: `.gitignore` already covers `*.key` patterns; verify after each commit with `git ls-files | grep -iE 'wpt.*key|api.*key' | wc -l` returns `0`.
- **Vector 4 — process listing leak:** Long-running `curl` with key in argv shows in `ps aux`. Mitigation: pass key via `-H` header constructed from `cat ~/.wpt-api-key` at invocation time; the header itself is in argv but only on this user's machine — acceptable on solo dev workstation per ASVS L1.

### Mitigations (acceptance-checkable)
1. Every WPT submission task passes the key ONLY via `-H "X-WPT-API-KEY: $(cat ~/.wpt-api-key)"` constructed inline; the key string is never stored in a shell variable that gets echoed elsewhere.
2. After each commit, `git ls-files | grep -iE 'wpt.*key|api.*key' | wc -l` MUST return `0` (verified in W0a acceptance, re-verified at every wave-end).
3. `chmod 600 ~/.wpt-api-key` is applied idempotently in W0a (verified via `stat -f '%A' ~/.wpt-api-key` returning `600` on macOS or `stat -c '%a' ~/.wpt-api-key` returning `600` on Linux).
4. No `set -x` or `curl -v` flags appear in any submitted command.
5. Test IDs (returned values from WPT API) MAY be logged — they are not credentials; the key MUST NOT.

### Validation
VALIDATION.md row `63-01-W0a / VRF-01 / secret-scan` — `git ls-files | grep -iE 'wpt.*key|api.*key' | wc -l` returns `0` — is the automated check.

### ASVS L1 fit
ASVS L1 sufficient — no new auth/session/crypto surfaces beyond the inherited WPT API key (reused from Phase 62 CONTEXT D-03). Severity raised from 62-01-PLAN's `low` to `high` for Phase 63 to match the explicit security_block_on=high gate; the underlying key handling is unchanged but the audit posture is stricter for gap-closure.
