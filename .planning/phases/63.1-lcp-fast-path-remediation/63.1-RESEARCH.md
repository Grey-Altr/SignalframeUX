### Recommended approach (3 sub-targets)

**1. `optimizePackageImports` extension (BND-02 carry-over)**

Current list: `["lucide-react", "radix-ui", "input-otp", "cmdk", "vaul", "sonner", "react-day-picker"]`. The `radix-ui` umbrella entry catches `radix-ui` (the meta-package) but NOT individual `@radix-ui/react-*` scoped packages — which is what `3302` and `7525` actually contain (per DGN-02). Extend to add scoped `@radix-ui/react-*` entries (select, menu, navigation-menu, scroll-area, toast, popper, tooltip, dialog, dismissable-layer, focus-scope) plus `date-fns` and `react-remove-scroll`.

Each addition's delta MUST be measured via `rm -rf .next/cache .next && ANALYZE=true pnpm build` (BND-04 + Pitfall #13). The 4 named chunk IDs MUST stay matched after the change.

**2. Homepage section route-level split**

`app/page.tsx:1-10` statically imports `EntrySection`, `ProofSection`, `ThesisSection`, `SignalSection`, `InventorySection`, `AcquisitionSection`. Only `EntrySection` is in the LCP viewport on first paint. The other 5 sections are below-the-fold but contribute their full GSAP/animation/component cost to First Load JS.

**Recommended split boundaries:**
- `EntrySection` — keep static (above-the-fold, contains CDB pointcloud rendering).
- `ProofSection`, `ThesisSection`, `SignalSection`, `InventorySection`, `AcquisitionSection` — convert to `next/dynamic({ ssr: true })` (NOT `ssr: false` — these need to be in the SSR HTML for SEO and the GhostLabel content-visibility intervention to keep working).

Why `ssr: true`: per Pitfall #3, lazy-mounting can cause LCP de-qualification if the LCP element ends up below a `display: none` boundary. SSR-rendered + JS-deferred is the safe pattern: HTML lands in initial response, JS chunk lazy-loads on `IntersectionObserver` proximity, hydration occurs after first paint. Next.js `next/dynamic({ loading: ..., ssr: true })` is documented as safe in App Router (as of Next 15 — verified via web search).

**Important caveat:** dynamic imports with `ssr: true` from a Server Component (`app/page.tsx` is currently a Server Component since it has no `'use client'`) work in Next 15.x but may not split as aggressively as expected — issue #61066 documents that some bundling shapes silently no-op the split. Wave 0 must verify the actual emitted chunk graph after the change with `ANALYZE=true pnpm build` and confirm the 5 below-fold sections moved out of `/`'s First Load JS.

**3. Homepage-tree GSAP lazy posture**

`584bde89-*` (50.5 KB GSAP core) currently lands in homepage First Load JS because `lib/gsap-core.ts` is imported by `lenis-provider.tsx` (top-level), `page-animations.tsx`, `scale-canvas.tsx`, plus 5 section files. Even though `LenisProvider`'s init body is rIC-deferred, the IMPORT happens at module-evaluation time, so the chunk lands eagerly.

**Recommended:** Convert section-level imports to dynamic. Pattern: `await import("@/lib/gsap-split")` inside the section's `useEffect`, with `cancelled` guard for unmount race. This pushes GSAP into the section's own async chunk, keeping it off the homepage First Load JS unless the section actually mounts. **Note:** this is an architectural change to 5 files (thesis, signal, proof, inventory, acquisition sections) and breaks `useGSAP`'s React-integration pattern (it's a hook, not a function). Plan 01 RESEARCH.md must verify whether `useGSAP` can be loaded async at all — if not, the fallback is to move only the GSAP module import to async while keeping `useGSAP` synchronously available; or accept that GSAP core stays on First Load JS and rely on the `optimizePackageImports` + section-route-split alone.

Reduce TBT from 317–3760 ms (real-device, varies by profile) to <300 ms even on slow Android. CPU-bound; cache-resistant; the dominant cost on Moto G Power 3G (TBT 3760 ms = 14.78× synthetic).

### Key inputs

- **Lenis CRT-04 pattern is canonical:** `components/layout/lenis-provider.tsx:60-68` shows the rIC-with-setTimeout-fallback pattern. Reuse this pattern; do not invent a new one.
- **Single-ticker rule (Pitfall #5):** the ONLY rAF call site in the project is `gsap.ticker` (which Lenis hooks into via `instance.raf`). New `requestAnimationFrame` calls are forbidden.
- **`requestIdleCallback` Safari status (2026):** caniuse / MDN confirm Safari support is rolling out (WebKit landed in 17.x behind feature flag; production-default on by ~Safari 18.4 per MDN compat data). The polyfill story is **don't polyfill via setTimeout queue** — use the inline `setTimeout(initLenis, 0)` fallback already in `lenis-provider.tsx:67`. This is the lightest-weight pattern, costs zero bytes (no polyfill package), and is what's already shipped. Per CONTEXT D-05 (no new runtime deps), this is the only acceptable posture.

### Recommended approach

**1. Section-level GSAP context idle-defer pattern**

For each of `thesis-section.tsx`, `signal-section.tsx`, `proof-section.tsx`, `inventory-section.tsx`, wrap the `useGSAP` initial setup in a rIC scope. The pattern follows CRT-04 verbatim: feature-detect `requestIdleCallback`, fall back to `setTimeout(cb, 0)`, and provide a cleanup that cancels the pending handle on unmount. **Critical:** the `useGSAP` hook itself MUST run synchronously (it's a React hook); only the *body* of its callback gets idle-deferred. Single-ticker rule preserved because all GSAP timeline / ScrollTrigger work still routes through `gsap.ticker` (which is the only rAF source).

**2. ScrollTrigger registration timing**

`gsap.registerPlugin(ScrollTrigger, …)` happens at module-evaluation time (`lib/gsap-core.ts:12`). This is fine because (a) it's idempotent, (b) Plan 01's section-level dynamic import will defer this to after first paint anyway. If Plan 01's GSAP-core lazy posture is rejected for `useGSAP` integration reasons, this remains a static cost on the homepage critical path.

**3. Polyfill posture for Safari (CONTEXT Claude's Discretion)**

**Recommendation: NO polyfill package. Use the inline pattern from CRT-04 / lenis-provider.tsx:64-68.**

Rationale:
- `requestidlecallback-polyfill` (1 KB minified) is a setTimeout-based shim that violates the spirit of the rIC API on Safari (no real idle detection).
- The inline `ric ? ric(cb, {timeout:100}) : setTimeout(cb, 0)` pattern is already shipped and proven (CRT-04). Adding a polyfill package to the dependency tree contradicts D-05 (no new runtime deps).
- Safari support is rolling out anyway; the fallback is genuinely a one-line setTimeout, and that's the canonical way to handle this in 2026.

## Plan 03: LCP fast-path — research findings + recommended approach

### Goal
Move the wordmark `<text>` paint off the JS-hydration critical path so all 3 profiles measure LCP <1000 ms. Synthesis §2 ¶3: "the wordmark `<text>` element should not block on the full JS payload."

### Critical clarification from late-milestone audit

**`cd-corner-panel.tsx` is ALREADY a pure Server Component.** It has zero `'use client'`, zero React hooks, zero state. The wordmark IS in the SSR HTML (verified by reading the file — 95 lines, all static JSX). So why is real-device LCP still failing?

The bottleneck is Chrome's **paint-after-blocking-JS-parse** sequencing. Even though `<CdCornerPanel />` is SSR'd, it's mounted under this hydration tree (`app/layout.tsx:155-170`):

```
<body>
  <WebVitals />            ← client component, but cheap
  …
  <TooltipProvider>        ← client wrapper
    <LenisProvider>        ← client wrapper, large
      <FrameNavigation />  ← client component
      <PanelHeightAssertion />
      <CheatsheetOverlay />
      <SignalframeProvider>     ← client provider
        <BorderlessProvider>    ← client provider
          <ScaleCanvas>         ← client wrapper, computes heights
            {children}          ← homepage
          </ScaleCanvas>
          <Nav />
          <NavRevealMount />
          <CdCornerPanel />     ← THE WORDMARK — server-rendered but inside hydration tree
        </BorderlessProvider>
      </SignalframeProvider>
    </LenisProvider>
  </TooltipProvider>
```

Chrome's LCP detector waits for the body subtree's blocking JS to finish parsing before it scores the paint of any element below the body's hydration boundary. The wordmark paint is gated by ~700 ms of JS parse on slow Android (per synthesis §2 finding 2). **The fix is to hoist the wordmark above (or outside) the hydration boundary.**

### Three SSR-decouple candidates (CONTEXT D-03 + Claude's Discretion)

**Candidate A — Server Component carve-out (move OUT of hydration boundary inside `<body>`)**

Move `<CdCornerPanel />` out from under `<TooltipProvider>` … `<BorderlessProvider>` and place it as a direct child of `<body>`, before any client provider opens (e.g. immediately after `<WebVitals />`).

Rationale: `CdCornerPanel` is already a Server Component; removing it from under the client-provider tree means React doesn't include it in the hydration root's render-after-JS-parse pipeline. Chrome paints it as soon as the SSR HTML is parsed, before blocking JS resolves.

| Pro | Con |
|-----|-----|
| Zero new code; pure HTML reorder | Loses visual layering with `<Nav />` if z-index isn't preserved (verify z-stack — `z-[var(--sfx-z-nav,40)]` is used; should compose fine) |
| Preserves `var(--sfx-cube-fill)` binding (already CSS-var driven, not React) | Loses theme-class scope if `dark` class is added to `<html>` after this — but `themeScript` runs first in `<head>` so `<html>` already has the right class before `<body>` parses |
| Preserves SVG `<mask>` knockout (it's all SVG) | `clipPath: "polygon(...)"` and `bottom: calc(var(--sf-frame-bottom-gap, 0px) + 24px)` rely on `--sf-frame-bottom-gap` which is set by `scaleScript` in `<head>` — already runs before this paint |
| Trademark primitive (D-12) preserved verbatim | Hydration mismatch risk: zero — Server Component, no JSX-stateful children |
| **Net: ZERO behavioral risk, maximum LCP win** | None identified |

**Candidate B — Static `.svg` asset emission + `<link rel="preload" as="image">`**

Pre-render the wordmark to a static `.svg` file at build time, place it at `public/cd-corner-panel.svg`, replace the React component with `<img src="/cd-corner-panel.svg" loading="eager" fetchpriority="high" />`, and add `<link rel="preload" href="/cd-corner-panel.svg" as="image">` in `<head>`.

| Pro | Con |
|-----|-----|
| Eliminates SVG from React tree entirely | LOSES `var(--sfx-cube-fill)` binding — theme-hue cube color is currently injected via CSS var, but `<img>` with static SVG can't pick up CSS vars. Static asset must hard-code the fill. |
| `<img>` can be eagerly loaded with `fetchpriority="high"` | Theme-aware (dark/light) wordmark would require 2 separate SVG assets + CSS swap, adding complexity |
| Smallest possible asset | LOSES the kana mask knockout interaction with page background — currently the kana glyphs cut through to the page content (true see-through per file comment); a static `.svg` rendered via `<img>` paints opaque against its element box, breaking the see-through aesthetic. |
| | **D-12 BLOCK risk:** trademark primitive's "see-through" knockout is part of the cube-tile trademark per AESTHETIC-OF-RECORD.md. A static asset that loses the see-through fails the trademark fidelity check. |
| | Build-time SVG generation requires a script; net new tooling for v1.8 (D-05 risk if dep added). |

**Candidate C — Pre-hydration inline `<svg>` in `app/layout.tsx` `<body>` (before React root mounts)**

Render the wordmark SVG markup *as a string literal* injected via the same `dangerouslySetInnerHTML` mechanism already used for `themeScript`/`scaleScript` (XSS-safe by construction — static literal, no user input — same posture as the already-shipped inline scripts). | Pro | Con |
|-----|-----|
| Maximum decoupling — markup paints at HTML parse time, before React root mounts | Hydration-mismatch risk: if React's tree includes the same `<CdCornerPanel />` (which it currently does), the two will conflict. Must remove `<CdCornerPanel />` from the React tree and rely solely on the inline injection. |
| `var(--sfx-cube-fill)` still works (inline SVG) | Loses TypeScript type-safety on the SVG markup (it's a string literal). |
| | Maintenance friction: any future change to the wordmark requires syncing two sources (the JSX + the string literal), unless we extract to a shared serializer. |

### Recommended approach (Plan 03)

**Primary: Candidate A — Server Component carve-out via `<body>` reorder.**

Implementation:
1. In `app/layout.tsx:170`, extract `<CdCornerPanel />` from inside `<BorderlessProvider>` and move it to `<body>` direct-child position, after `<WebVitals />` but before the providers tree opens.
2. Verify z-stacking: `z-[var(--sfx-z-nav,40)]` currently composes with `<Nav />`; after the move, ensure stacking context is preserved (should be — both are now `<body>` children).
3. Verify `<dark>` class scope: the `themeScript` (line 108) writes to `document.documentElement` (i.e. `<html>`), which is an ancestor of the moved `<CdCornerPanel />`, so theme-class still applies.
4. Verify `--sf-frame-bottom-gap` and `--sf-frame-offset-x` propagation: `scaleScript` writes to `document.documentElement.style`, which cascades to all descendants regardless of provider tree shape.
5. Keep `<link rel="preload">` deferred to a **stretch goal** — first measure LCP delta from the carve-out alone. If A alone closes the gap (synthesis §2 names ~700ms of blocking JS parse as the primary cost; carving the wordmark out of the hydration boundary should reclaim most of this), no preload needed.

**Fallback: Candidate C — pre-hydration inline `<svg>` injection.**

Trigger condition: if Candidate A's LHCI delta is <300 ms and post-WPT real-device LCP still >1000 ms on iPhone 14 Pro, escalate to C. The inline injection guarantees paint at HTML-parse time and is the strongest possible decoupling short of static assets.

**Rejected: Candidate B.** D-12 trademark fidelity gate fires on the see-through knockout loss. Document the rejection in Plan 03 RESEARCH.md per `_path_X_decision` pattern if measurement later forces it back into consideration.
