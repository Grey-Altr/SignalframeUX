<threat_model>
ASVS L1 — Phase 64 Plan 01 — Block on HIGH.

T-64-01 (HIGH) — Branch protection bypass via misconfigured required-check name.
  Attack: required-check entry saved BEFORE first LHCI run completes, or with mistyped check
          name. GitHub silently saves the rule with NO required checks; rule appears configured
          but offers zero protection.
  Mitigation: Strict task ordering enforced — Task 1 (deployments:write) → Task 2 (trigger
              first LHCI run with noop PR) → Task 3 (add `audit` to required checks). Task 3
              acceptance criterion is a `gh api` probe that asserts `"audit"` is literally
              present in `.required_status_checks.contexts[]`.

T-64-02 (HIGH) — GitHub App over-permissioning.
  Attack: granting broader scope than `deployments:write` on the Vercel GitHub App
          (e.g., `contents:write`, `actions:write`, `secrets:write`) expands the blast
          radius if Vercel is ever compromised.
  Mitigation: Task 1 instructions explicitly list `Deployments: Read & write` as the ONLY
              permission to add. Task 1 acceptance criterion includes a screenshot/textual
              audit of current Vercel App permissions to confirm no scope creep.

T-64-04 (LOW) — Spec recalibration weakens future gate. (Out of scope for this plan;
              addressed in Plan 02's threat model.)

Block-on threshold: HIGH. T-64-01 and T-64-02 both have explicit mitigations encoded in
task ordering and acceptance criteria.
</threat_model>
