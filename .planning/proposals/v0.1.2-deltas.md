# SignalframeUX Manifest v0.1.2 — Proposed Deltas

**Derived from:** `feature/navbar-redesign` branch (64 commits ahead of main as of 2026-04-18)
**Supersedes:** `design.md` v0.1.1 where noted
**Status:** Draft — not yet merged into `design.md`

## Why these deltas

v0.1.1 locked down the system against Claude Design v0 probe drift (canonical ease-out, explicit rejected curves, no-subsystem-invention rule, hex ban reinforcement). Since then `feature/navbar-redesign` has authored expressive-layer systems that v0.1.1 doesn't sanction but are not drift — they are intentional SIGNAL-layer growth. This document captures what to canonize, what to clarify, and what remains code-level bugs (not manifest material).

Three categories:

- **§A Sanctions** — new subsystems and token families to add to §4 and clarify §3's "no subsystem invention" rule.
- **§B Clarifications** — tighten ambiguities that created the drift surface; no new systems, just sharper language.
- **§C Still-to-fix** — authoring mistakes on `feature/navbar-redesign` that are NOT manifest material. Fix in code, do not canonize.

## §A — Sanctions (new sections for design.md)

### A.1 Effects Pipeline (new §4.10)

Already hinted at in §5 "Effects Subsystem → `SFSignalComposer`" but the token contract is undefined. Canonize:

| Token | Default | Purpose |
|---|---|---|
| `--sfx-fx-tier` | `fallback` | Current effects tier (`fallback` / `baseline` / `enhanced` / `premium`) |
| `--sfx-fx-multiplier` | `0` | Master scalar for all effect-layer opacity/intensity (0–1) |
| `--sfx-fx-feedback-decay` | `0.88` | Feedback loop decay coefficient |
| `--sfx-fx-displace-gain` | `0` | Displacement field gain (0–1) |
| `--sfx-fx-bloom-intensity` | `0` | Bloom pass intensity (0–1) |
| `--sfx-fx-glitch-rate` | `0` | Glitch pass event rate (0–1) |
| `--sfx-fx-particle-opacity` | `0` | Particle field opacity (0–1) |
| `--sfx-audio-bass` | `0` | Audio-reactive bass amplitude (0–1) |
| `--sfx-audio-mid` | `0` | Audio-reactive mid amplitude (0–1) |
| `--sfx-audio-high` | `0` | Audio-reactive high amplitude (0–1) |

**Contract:** These tokens are written at runtime by `updateSignalDerivedProps()` in `components/layout/global-effects.tsx`. CSS consumers may read them without JS. Primitives in `components/animation/` use the JS API in `lib/effects/` directly for numeric precision.

**Does NOT expand the component surface.** These tokens do not spawn new CVA variants or new SF components. They parameterize existing effect primitives.

### A.2 Canvas Frame System (new §4.11)

The ScaleCanvas aspect-lock contract that fixed-position chrome must consume. Pre-existing in runtime, absent from v0.1.1 manifest.

| Token | Default | Purpose |
|---|---|---|
| `--sf-vw` | `1vw` (no-canvas fallback) | Canvas-remapped viewport width unit |
| `--sf-vh` | `1vh` (no-canvas fallback) | Canvas-remapped viewport height unit |
| `--sf-canvas-scale` | `1` | Active canvas scale factor |
| `--sf-canvas-h` | `calc(100 * var(--sf-vh))` | Canvas height in CSS pixels |
| `--sf-frame-offset-x` | `0px` | Horizontal offset of canvas within viewport |
| `--sf-frame-bottom-gap` | `0px` | Bottom-edge gap (for fixed HUD clearance) |
| `--nav-height` | `83px` | Top nav fixed height (consolidates with `--sfx-nav-height`) |

**Contract:** Set by `components/layout/scale-canvas.tsx`. Fixed chrome (nav, HUDs, toasts) must consume these rather than `vw`/`vh` directly so that scaled / letterboxed canvas states remain coherent. Naming exception: these use the `--sf-*` prefix (not `--sfx-*`) because they are runtime viewport vars, not token values.

### A.3 Keyframe Library (new §4.12)

v0.1.1 §3 says "No new GSAP effects." It does not address CSS `@keyframes`. Canonize the existing named keyframe set so future additions have a ceiling.

**Permitted keyframe families (defined in `app/globals.css`):**

| Family | Keyframes | Purpose |
|---|---|---|
| Fade | `sf-typewriter-fade`, `sf-slow-fade-in`, `sf-hero-slow-fade-in`, `sf-feel-bloom-fade-in` | Intro/reveal timing |
| Nav | `sf-nav-roll-up`, `sf-nav-icon-hover-hold` | Nav chrome micro-behavior |
| Hero | `sf-hero-slash-enter`, `sf-period-in` | Display-type entries |
| VHS/CRT | `sf-vhs-ease-in`, `sf-scanline-drift`, `sf-idle-scan`, `sf-jfm-flicker` | Signal-layer overlay motion |
| Feedback | `sf-skeleton-sweep`, `sf-shake`, `sf-flash`, `sf-feel-pulse` | User-feedback micro-motion |
| Mesh | `sf-mesh-drift` | Background generative motion |
| Marquee | `sf-marquee-scroll` | Horizontal ticker |

**Addition policy:** New keyframes require a family assignment. Keyframes outside these families count as subsystem invention and go through the proposal workflow. All keyframes must use the canonical ease-out curve or a monotonic math-function progression (see A.5).

### A.4 HUD Micro-Type Scale (§4.3 addendum)

v0.1.1 §4.3 defines scale from `--text-2xs` (9px) to `--text-4xl` (80px). Usage analysis shows HUD chrome and miniature component previews need sub-9px sizes.

Add:

| Token | Target @1440 | Use |
|---|---|---|
| `--text-hud-xs` | 5px | Miniature preview pixel grid labels |
| `--text-hud-sm` | 6px | Miniature preview inline chrome |
| `--text-hud-md` | 7px | Compact chrome rows in mini-previews |
| `--text-hud-lg` | 8px | HUD badges, footer pill text |

**Constraint:** HUD micro-type is for CHROME ONLY — never for content-carrying copy. Accessibility exception is explicit: these tokens fall below WCAG minimum readability and must not render text that users are expected to read. Applies to decorative previews, glyph chrome, instrumentation HUDs, and inline meta labels where the size IS the aesthetic signal (Ikeda perception-threshold register).

Additionally: `text-[11px]` usages (HUD labels in `reference/`, `inventory/`, `system/` pages) should snap to `--text-sm` (11px per §4.3) with a build check. No new token needed.

### A.5 Procedural SIGNAL-Layer Motion (§4.5 addendum)

v0.1.1 §4.5 says all easing tokens resolve to `cubic-bezier(0, 0, 0.2, 1)`. This applies to TRANSITIONS. It does not address PROCEDURAL motion driven by CSS math functions (`pow()`, `sin()`, runtime-written custom properties).

Clarify: parametric animations that compute motion progression via CSS math on a canonical linear scrub variable are allowed to use non-bezier progression functions. The nav cube cascade (`--sf-nav-peel-frac: calc(1 - pow(calc(1 - var(--sf-nav-phase-frac)), 6))`) is the reference pattern. Restrictions:

1. Must operate on a monotonic input (scrub progress, scroll position, time); no bouncing inputs.
2. Output range must stay within `[0, 1]` for t in `[0, 1]`. No overshoot.
3. Documented per component. Not a reusable math utility layer.

This is NOT a license to reintroduce bezier overshoot curves.

### A.6 Structural Gradients (§3 Hard Constraints — clarification)

v0.1.1 §3 says "Gradients — None." Observed navbar-redesign usage reveals gradients used as SHAPE PRIMITIVES (not decoration): solid-stop bars for hero text-strikes, edge-vignette masks for scroll clipping, scanline patterns for VHS overlay. Rewrite §3's gradient rule as:

> **Gradients** — No decorative gradients. Permitted structural uses:
> - **Hard-stop bars** — `linear-gradient(var(--sfx-primary) 42%, transparent 42%, ...)` for typographic strike marks. No soft color transitions.
> - **Edge vignettes** — `mask-image: linear-gradient(to bottom, transparent 0%, black 30%, black 70%, transparent 100%)` for scroll edge clipping. Used only as `mask-image` / `-webkit-mask-image`.
> - **Scanlines** — `repeating-linear-gradient(...)` for CRT/VHS overlay. Limited to the signal-layer overlay subsystem.
> - **Halftone / grid** — `repeating-linear-gradient` or equivalent with hard color stops for structural texture in SIGNAL-layer panels.
>
> Forbidden: decorative blur-to-color gradients, aurora backgrounds, glassmorphic gradients, rainbow, color-shift ornamentation, generic "mesh" gradients.

### A.7 Edge-Runtime OKLCH Exemption (§3 / §7 addendum)

v0.1.1 §3 bans hex colors anywhere in output. The Edge Runtime (Satori for `opengraph-image.tsx`, Next.js `icon.tsx`, Edge-function `global-error.tsx`) cannot resolve CSS custom properties — tokens must be inlined as literal values, and the Satori renderer does not accept `oklch()` strings in all contexts.

Clarify: Edge-runtime files may use hex color literals **only when** the value is sourced from a compile-time constant map derived from the OKLCH token definitions. Free-authored hex remains forbidden.

Proposed pattern: add `lib/tokens.edge.ts` exporting `EDGE_COLORS` as a `Record<TokenName, string>` computed from the canonical OKLCH definitions via a build step or static mapping. Edge files import from this module; no inline hex in the source.

Scope: applies ONLY to files rendered by Edge Runtime / Satori. Server Components, Client Components, and CSS modules remain under the OKLCH-only rule.

### A.8 Canvas 2D Fallback Pattern (§3 addendum)

Canvas 2D contexts (particle fields, displace fields, WebGL shader inputs) cannot consume CSS custom properties directly. Current pattern in `components/animation/particle-field-hq.tsx`:

```ts
const computed = getComputedStyle(document.documentElement).getPropertyValue("--sfx-primary");
const pColor = particleColor.includes("var(") ? computed || "#ff00ff" : particleColor;
```

Clarify: Canvas contexts may use `getComputedStyle()` to read OKLCH tokens and convert to hex/RGB at runtime. A literal hex fallback is allowed as last resort ONLY when it matches the default theme hue magenta (`oklch(0.65 0.3 350) ≈ #FF0090`). Component authors should use the `EDGE_COLORS.primary` value from A.7 rather than free-authoring hex.

## §B — Clarifications (tightenings, no new systems)

### B.1 Easing authoring discipline (§4.5 reinforcement)

v0.1.1 §4.5 forbids CSS named easings, rejected curves, and multiple ease-out variants. Observed drift on navbar-redesign:

- `ease-out` / `linear` / `ease-in-out` appear inline in `globals.css` ~30+ times.
- `cubic-bezier(0.22, 1, 0.36, 1)`, `cubic-bezier(0.16, 1, 0.3, 1)`, `cubic-bezier(0.16, 0, 0.5, 1)` — three alternate ease-out variants.

Add to §4.5: "When authoring a transition or CSS animation, references must be to the token (`var(--sfx-ease-default)` et al.) OR to the literal canonical string `cubic-bezier(0, 0, 0.2, 1)`. Any other literal bezier or named easing in an animation/transition context fails compliance."

Enforcement: add a grep-based CI check.

### B.2 Keyframe easing consistency (§4.5 reinforcement)

Each `@keyframes` definition has implicit easing via the consumer's `animation-timing-function`. All consumers must use the canonical ease-out (or a SIGNAL-layer procedural progression per A.5). Reject named easings inside `animation:` shorthand.

### B.3 Spacing scale at non-anchor viewports (§4.2 clarification)

v0.1.1 §4.2 defines blessed stops via `clamp()` with a 1440px anchor. Between `clamp` floor and ceiling, rendered values fall continuously and may not equal a listed stop. Clarify:

> The blessed stops are anchor values at 1440px. The `clamp()` expressions produce scaled-but-proportional values at other viewports. Component spacing MUST reference `--sfx-space-N` tokens; never author raw pixel spacing that would "aim for" an intermediate value.

This closes the ambiguity of whether, e.g., `--sfx-space-12` clamping to 72px at some viewports is a violation. It is not — the scale is token-anchored, not pixel-anchored.

### B.4 `--sfx-ease-spring` name deprecation (§4.5 reinforcement)

Per v0.1.1 §4.5: "`--sfx-ease-spring` is a legacy name; the curve is ease-out." Deprecate the token name. v0.1.2 replaces `--sfx-ease-spring` consumers with `--sfx-ease-default`. Remove from published token list. Keep as alias for one version with a `@deprecated` comment in `globals.css`.

### B.5 Viewport-unit consumption policy (§4.4 addendum)

Components inside `<ScaleCanvas>` should consume `--sf-vw` / `--sf-vh` instead of `vw` / `vh`. Components OUTSIDE `<ScaleCanvas>` (global chrome, modals, portals) consume standard viewport units OR tokens from §4.4. Add a note to §4.4 establishing this dependency direction.

## §C — Still-to-fix (code bugs, NOT manifest material)

These are authoring mistakes on `feature/navbar-redesign`, not design decisions. Canonizing them would be incorrect.

### C.1 Malformed Tailwind arbitrary value

Pattern: `px-[var(--sfx-space-1)].5` / `py-[var(--sfx-space-1)].5` — 9 occurrences across:
- `components/blocks/components-explorer.tsx` (8×, lines 237, 247, 248, 259, 288, 292, 295, 383, 384, 385, 414)
- `components/animation/signal-overlay.tsx` (1×, line 273)

The `.5` after `]` is outside Tailwind arbitrary-value syntax. It either no-ops or produces a stray ".5" class with no effect. Intent was probably `px-1.5` (6px), which isn't a blessed stop anyway.

**Fix:** drop the `.5` to match the runtime-observed behavior: `px-[var(--sfx-space-1)]`. If visual reveals spacing is too tight, escalate to `--sfx-space-2`.

Introduced in `c1fb678`.

### C.2 Raw pixel spacing in WIP

`app/globals.css:654` (WIP): `--sf-nav-cube-pitch: calc(32px + 3px);` replaces the committed `calc(32px + var(--sfx-space-1))`. The 3px literal is not on the blessed scale.

**Fix:** revert to token reference, or if the 3px is specifically load-bearing for the cascade math, justify inline.

### C.3 Raw overshoot / anticipation bezier curves

`app/globals.css` has ~14 occurrences of `cubic-bezier(0.34, 1.56, 0.64, 1)` (overshoot) and one `cubic-bezier(0.68, -0.2, 0.27, 1.2)` (anticipation) in navbar transitions. Both are in §4.5's rejected-curves list.

**Fix:** refactor all transitions to `cubic-bezier(0, 0, 0.2, 1)` or `var(--sfx-ease-default)`. Expected visual change: springy hovers become decisive. Consistent with v0.1.1 §4.5 intent.

Locations (non-exhaustive): `globals.css:1396, 1399` (nav underline), `:1009` (theme toggle knob), `:2271, 2329, 2345, 2361, 2373, 2402, 2431, 2464` (nav cube & transitions).

### C.4 Alternate ease-out bezier variants

`cubic-bezier(0.22, 1, 0.36, 1)` (7×), `cubic-bezier(0.16, 1, 0.3, 1)` (2×), `cubic-bezier(0.16, 0, 0.5, 1)` (1×) — all in `globals.css`. §4.5 forbids multiple ease-out variants.

**Fix:** collapse to canonical `cubic-bezier(0, 0, 0.2, 1)`.

### C.5 CSS named easings in animations

`ease-out`, `linear`, `ease-in-out` appear inline in `globals.css` across 30+ `animation:` and `transition:` declarations. §4.5 forbids named easings.

**Fix:** replace with explicit canonical bezier or `var(--sfx-ease-default)`.

### C.6 In-CSS hex

`app/globals.css:711` (`background-color: #fff`) and `:2214` (`color: #666 !important`). §3 bans hex in output.

**Fix:** replace with OKLCH equivalents (`oklch(1 0 0)` and `oklch(0.55 0 0)` respectively, to be verified for contrast).

### C.7 `lib/tokens.css` easing drift

`lib/tokens.css:214–216` still defines `--sfx-ease-hover: cubic-bezier(0.34, 1.56, 0.64, 1)` (rejected overshoot) and `--sfx-ease-spring: linear(...)` (rejected spring-linear). This file post-dates v0.1.1 and must be brought into compliance: all three easings flatten to `cubic-bezier(0, 0, 0.2, 1)`.
